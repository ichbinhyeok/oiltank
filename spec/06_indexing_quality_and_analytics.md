# 06 Indexing Quality And Analytics

## Indexing philosophy
Index only pages that are:

- scenario-specific
- state-aware
- source-backed
- commercially meaningful

## Promotion recommendation principle
Held routes should not auto-promote just because they exist in the system.

The correct loop is:

1. metrics arrive
2. the agent reviews route evidence
3. the agent writes a recommendation
4. the user approves or rejects the promotion

This protects the wedge while removing the need for the human to remember every project.

## Pages that should index first
- home
- five state hubs
- five state buyer-seller pages
- five state sweep pages
- five state records pages
- strongest evergreen guides

## Pages that should stay `noindex` at first
- thin cost pages
- speculative county overlays
- partner finder pages
- weak market pages with no local official difference

## Route promotion rules
A route can move from `noindex` to `index` only when:

1. official source support is clear
2. the route materially differs from broader pages
3. the page has a realistic CTA and commercial path
4. the route has at least one strong internal-link source

## Event taxonomy
Track at minimum:

- `page_view`
- `route_cta_click`
- `guide_to_state_click`
- `lead_form_open`
- `lead_submit_attempt`
- `lead_submit_success`
- `lead_submit_error`
- `partner_outbound_click`

## Early dashboard metrics
- indexed pages
- impressions by route family
- CTR by route family
- CTA clicks by route family
- lead submissions by scenario
- lead submissions by state

## Promotion evidence inputs
Every promotion review should look at:

- search impressions and clicks by indexed parent route
- query evidence pointing to a held route need
- CTA clicks by state and scenario
- lead submissions by scenario
- source freshness
- partner availability for the route's commercial action

## First proof-of-traction metrics
- impressions on state buyer-seller pages
- clicks on sweep or records-checklist CTAs
- lead submissions from `NJ`, `NY`, or `CT`

## Held-route recommendation rules

### Removal-versus-abandonment route
Recommend only when:

- the state's indexed buyer-seller, sweep, or records pages show repeated confirmed-tank or next-step ambiguity
- source freshness is green
- the route has a real CTA path beyond generic quote bait

### Leak-and-cleanup route
Recommend only when:

- lead submissions or admin notes show real leak-concern frequency
- the state's source stack supports reporting and cleanup guidance clearly
- the route can safely separate suspicion from actual contamination workflow

### Cost-direction route
Recommend only when:

- indexed routes are already earning clicks and scenario evidence
- directional cost language is source-backed for that state
- the route can stay decision-first instead of becoming a thin price page

### Market overlay route
Recommend only when:

- the state route already proves demand
- the market materially changes the answer
- the market has enough commercial density to justify maintenance

## Agent review cadence
- every meaningful implementation or review session after metrics exist
- every `14` days after launch, even if no route is promoted

If no route qualifies, the agent should still record `hold` with a reason so the user does not have to remember what was checked.

## Quality gates
- no page should be indexed without source freshness dates
- no route family should expand before the first route in that family earns impressions
- if a state route has no meaningful unique answer, merge it into a guide or keep it out of the index

## Kill rules
Kill or demote any route family that:

- attracts traffic but no action
- duplicates the state hub without new value
- depends mostly on weak non-official sources
- drifts into broad informational intent

## Analytics principle
The point of analytics is to learn which trigger state converts, not to admire pageviews.
